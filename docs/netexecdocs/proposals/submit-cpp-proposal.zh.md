# 如何向 ISO C++ 提交提案

> 原文：https://isocpp.org/std/submit-a-proposal

本文面向想给 C++ 标准库提交新功能提案的人。核心语言特性提案门槛更高，见文末 FAQ。

## 提交流程（三步）

### 第一步：抛出想法（Float the idea）

在 **std-proposals 邮件列表** 发一个简短的描述，重点说明：

- 你的功能解决什么问题
- 考虑过哪些替代方案

委员会成员和专家会给出反馈，帮你判断这个方向是否值得继续做，以及你的思路有没有明显问题。

> 需要订阅邮件列表才能参与讨论，但 archives 可以匿名浏览。

### 第二步：发布初稿（Post an initial draft）

用页面底部的**提案模板**写一份初稿，在封面上加一段 **Abstract**（1-2 段，概述问题和方法），然后把链接发到 std-proposals 邮件列表，开始第二轮讨论。

这一版应该比第一步更详细，能让别人给具体修改意见。

### 第三步：迭代打磨（Iterate and fine-tune）

根据反馈不断更新草案、发布新版本，重复几轮，直到设计获得较广泛支持。

一旦大家在"需要解决这个问题"和"具体设计方案"上开始趋同，就可以恭喜你：这份提案已经可以作为正式编号论文提交，安排在某次 ISO C++ 标准会议上由你或你的合作者亲自介绍。

## 常见问题（FAQ）

### Q1: 提案必须有实现吗？

**不是 ISO 的硬性要求。** 关键是达成共识。但 naturally，如果有可用实现，更容易获得共识。

委员会成员对"现有实践"的重视程度是个谱：
- 一端是简单到仅凭提案本身审查就能接受；
- 另一端是复杂到需要完整实现、多年广泛使用、跨平台验证后才可能接受。

每个委员对具体提案落在哪一端有独立判断。

### Q2: 知识产权方面要注意什么？

如果提案中可能包含你自己、雇主或他人的专利/版权内容（包括你在公开讨论中看到的想法），请确保阅读并遵守 **ISO Directives and Policies**，包括专利政策。

只有 ISO 能最终解释这些政策，具体问题请联系 ISO Central Secretariat。

### Q3: 如何向委员会展示提案？

1. **先和相关 subgroup/study group 的 chair 预约时间。** 提案需要由提案者（或非常熟悉该提案的同事/代表）亲自介绍；没人介绍的提案通常不会被考虑。联系 committee page 上合适的 chair 安排会议时间。如果不确定该找谁，先联系任意一位 committee officer 或 subgroup chair，他们会把你转给正确的人。

2. **准备一段简短的演讲。** 你是这个领域的专家，要向一群 C++ 专家介绍。不要假设大家都仔细读过论文——他们多数提前看过，但实时交流还是不同的。演讲目标是：
   - 让大家快速进入状态
   - 回答关于论文的问题
   - 推销关键设计点

   建议覆盖：
   - **Motivation**：什么问题？为什么重要？优先级？为什么这样解决？
   - **Design tradeoffs**：A/B/C 各有什么优缺点，为什么选择 B？
   - **Open questions**：还没决定的设计点、需要性能数据的地方、实现可行性等。

3. **要有耐心。** 成功的提案通常需要**多次标准会议**。第一次介绍时大家可能还在消化，看起来犹豫不决是正常的。等大家"get it"之后，才会逐渐形成共识。

### Q4: 如果我想写提案但无法参会怎么办？

你需要找一个能参会、能全面代替你的人。这个人必须：
- 非常熟悉论文
- 支持这个提案，像作者一样理解它
- 必要时能接管并成为后续版本作者

### Q5: 如何提案 C++ 核心语言特性？

核心语言特性提案门槛**高得多**，无论技术上还是战略上。强烈建议先至少亲自参加一次委员会会议。经验表明，如果不这样做，很容易把时间花在"还没烤熟"或与 C++ 语言方向不一致的提案上。会议对 observer 和新参与者是开放的。

## 标准库提案模板

以下模板适用于中等到高复杂度的标准库功能提案。简单提案可以适当简化。

> **Document number:** PnnnnR0  
> **Date:** yyyy-mm-dd  
> **Audience:** Library Evolution Working Group  
> **Reply-to:** Your name and email address（可以写成 "Jane Proposer <jane at somewhere dot com>" 这种混淆格式）

### I. Table of Contents

目录

### II. Introduction

非常简短的高层概述，让不熟悉该领域的委员会成员也能理解。

### III. Motivation and Scope

- 为什么重要？
- 解决什么问题？
- 目标用户群体是谁？
- 面向什么水平的程序员？
- 基于什么现有实践？使用范围多广？用了多久？
- 是否有参考实现和测试套件可供查看？

### IV. Impact On the Standard

- 依赖哪些标准库组件？
- 哪些组件会依赖它？
- 是纯扩展，还是需要修改现有组件？
- 能否用当前 C++ 编译器和库实现？是否需要当前 C++ 没有的语言/库特性？

### V. Design Decisions

- 为什么选择这个设计？
- 考虑过哪些替代方案？各自的 tradeoff 是什么？
- 对用户和实现者有什么影响？
- 哪些决策留给实现者？
- 如果有类似的库在用，它们的设计决策和你的有什么不同？

### VI. Technical Specifications

委员会需要完整的技术规范来评估。最终需要写成"Standardese"（标准措辞），但初稿可以分几种情况：

- **有限技术文档**：适用于非常简单的提案（如单个函数），但委员会通常会要求更多。
- **完整技术文档**：足够让人全面评估。可以放在提案里，也可以给外部链接。如果委员会喜欢，会要求下一版给出正式标准措辞。委员会认识到写 ISO 规范有难度，会提供额外帮助。
- **完整 Standardese**：直接写成标准合同式措辞，明确实现者可以做什么、必须做什么、用户可以依赖什么。风格应符合 Specification Style Guidelines，但不需要完全匹配页边距、字体、章节编号——这些之后都会改。
